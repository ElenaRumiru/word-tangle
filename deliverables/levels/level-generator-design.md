# Логика генератора уровней (фаза 4)

Deliverable фазы 4. Пара к `levels-design.md` (кривая, формула скрэмбл-скора, инвариант
«каждое слово отдаёт ≥1 букву»), `validate_levels.py` (гейт), `redistribute_circles.py`
(подбор кружков как задача о назначениях). Контент — английский, обоснование — русское.

Цель документа — не «в общих чертах», а **инженерный чертёж** конвейера, который выпускает
**тысячи уровней** с сложностью, попадающей в целевую кривую фазы 5. Ниже — каждый шаг со
входами/выходами/инструментами, честное узкое место, числа пропускной способности и стоимости,
и proof-of-concept (`poc_word_search.py`), эмпирически показывающий, что пространство решений
огромно.

---

## 0. Архитектура: что генерируется автоматически, а что нет

Ключевое различие, которое определяет весь дизайн:

| Слой | Генерируется комбинаторикой? | Кто узкое место |
|---|---|---|
| **Ответ-каламбур** (`"WADE" A MINUTE`, `PANE RELIEF`) | **Нет.** Это творческая единица — игра слов, привязанная к омофону/идиоме и к сцене. Комбинаторика не породит «pane=pain». | **ДА — главный боттлнек.** LLM-кандидаты + ручная модерация. |
| **Слова под кружки** (какие 4–5 слов отдадут буквы) | **Да.** Чистый CSP над словарём: покрыть мультисет букв ответа. Решений — миллиарды (см. §3). | Нет. Тысячи/сек на ноутбуке. |
| **Скрэмблы** | **Да.** Перестановки под целевой P4. Дёшево. | Нет. |
| **Картинка-карикатура** | Полу-авто: шаблонный промпт → image-API → vision-проверка. | **Вторичный боттлнек** (деньги + ревью). |
| **Валидация / скоринг / раскладка** | **Да.** Детерминированные проверки, `validate_levels.py`. | Нет. |

**Вывод для стратегии:** генератор — это не «одна кнопка», а **воронка**, где дорогой ручной
ресурс (модерация каламбуров) тратится только на верхний слой, а всё остальное автоматика
достраивает вокруг уже одобренного ответа. Это позволяет 1 человеку выпускать сотни уровней/день.

Порядок конвейера намеренно ставит **банк каламбуров первым**: сначала одобряем дорогой ответ,
потом дешёвая автоматика строит под него слова/скрэмблы/картинку. Строить слова, а потом искать
каламбур под них — значит тратить дешёвый ресурс до дорогого гейта. Воронка течёт от узкого к широкому.

## 1. Корпуса данных (вход конвейера, строится один раз)

Инструменты и артефакты, которые кэшируются и переиспользуются всеми уровнями:

| Корпус | Источник | Формат в рантайме | Роль |
|---|---|---|---|
| **Частотный словарь** | `wordfreq` (EN: SUBTLEX/OpenSubtitles Zipf-ранги); для RU-версии — НКРЯ/`wordfreq ru` | `word → zipf`; индекс `(band, len) → [words]` | P3-банды сложности, гейт «ответ — реальное слово» (d), гейт «скрэмбл — не слово» (b) |
| **Стоп-лист** | ручной + опенсорсные списки | `set[str]` | профанити, слэры, бренды/торговые марки — фильтр на ответах И на скрэмблах |
| **Банк идиом** | Wiktionary idioms + курируемый список | `[{idiom, meaning, tags}]` | сырьё для каламбуров типа `OLD HAT`, `PIER PRESSURE` |
| **Банк омофонов** | CMUdict (совпадение фонем) + ручная чистка | `phoneme → [words]` | сырьё для `HEIR/AIR`, `WADE/WAIT`, `PANE/PAIN` |

Zipf-банды (совпадают с P3 из `levels-design`):

```
top  : 5.0–9.0   (L1-ish, «NAP», «TREE»)
mid  : 4.0–5.0
low  : 3.0–4.0
rare : 2.5–3.5   (L100-ish, «POROUS», «SLEIGH», «TETHER»)
```

Индекс `(band, len) → [words]` — то, из чего CSP §3 набирает слова. В PoC он строится за **0.29 с**
из 28 647 слов длиной 4–7 (16 бакетов). Строится один раз, дальше он в памяти.

**Выход шага:** словарь Zipf, стоп-лист, банк идиом/омофонов — всё в памяти/на диске, переиспользуется.

## 2. Банк ответов-каламбуров — ЧЕСТНОЕ узкое место

**Это самое дорогое и самое медленное место конвейера, и это надо признать прямо.** Каламбур —
не комбинаторный объект: `PANE RELIEF` работает, потому что pane=pain (омофон) *и* сцена — доктор,
у которого «отлегло». Ни один генератор перестановок это не выведет. Значит — цепочка
«LLM-кандидаты → автофильтры → человек».

**Пайплайн банка:**

```
Тема/тип/длина ──LLM батч──▶ N кандидатов ──автофильтр──▶ M чистых ──человек──▶ одобренные
   (промпт)       (дёшево)                   (дёшево)                (ДОРОГО)
```

1. **LLM-батч.** Промпт-шаблон:
   > «Дай 40 каламбурных ответов-сюрпризов в стиле Jumble для типа `homophone`,
   > длина финала 8–12 букв, тема `sports`. Для каждого верни: `answer`, `setup_caption`
   > (подводка с многоточием), `pun_word`, `pun_type`, короткое обоснование гэга.»

   Один вызов = десятки кандидатов. Стоимость — копейки (см. §9).

2. **Автофильтры (детерминированные, бесплатные):**
   - длина финала в целевом диапазоне;
   - `pun_word` реально в словаре / банке омофонов (zipf-проверка);
   - дубликат по уже одобренным (нормализованный ключ answer);
   - профанити/бренд в ответе → выброс;
   - грубая проверка «это вообще каламбур» (совпадение фонем pun_word с исходным словом через CMUdict).

   Фильтры срезают ~50–70% сырья, оставляя человеку только правдоподобное.

3. **Ручная модерация (боттлнек).** Человек смотрит отфильтрованный топ и отвечает на 3 вопроса:
   смешно? честный ли каламбур (не притянут)? семейно-безопасно? Реалистичный темп —
   **~40 одобренных ответов/час** (≈1.5 мин на кандидата с учётом отсева; см. расчёт §8).

**Как масштабируется:** линейно по модераторам — 2 человека дают ×2 банка. LLM и автофильтры
масштабируются практически бесплатно, поэтому модератор всегда «сыт» кандидатами. Банк —
**переиспользуемый актив**: одобрил 500 каламбуров один раз — генератор строит из них уровни
месяцами. Именно поэтому годовая амортизация делает узкое место терпимым.

**Выход шага:** таблица одобренных каламбуров `[{answer, setup_caption, pun_word, pun_type, theme}]`.

## 3. Подбор слов под кружки — CSP (дёшево, решений миллиарды)

Вход: одобренный `answer` (мультисет букв без пробелов) + целевые `K` (число слов, = P1),
длина слов (P2), Zipf-банд (P3) из строки балансовой таблицы.

Задача — **ровно инвариант фазы 1**: выбрать `K` слов заданной длины/банда так, что
1. пул их букв покрывает мультисет ответа,
2. **каждое слово отдаёт ≥1 букву** (authentic-Jumble, гейт (g)),
3. распределение сбалансировано (1–3 кружка/слово).

Это constraint-satisfaction. Двухфазный решатель:

**Фаза A — выбрать K слов (набор-покрытие).** Жадность + backtracking по индексу
`(band,len)→[words]`, с кэшом «буква → слова банда с этой буквой»:

```
select_words(answer, K, band, len):
    need = Counter(answer)                      # мультисет, который надо покрыть
    pool = index[(band, len)]
    solution = []
    def bt(remaining_need, words_left):
        if words_left == 0:
            return remaining_need.empty()        # все буквы покрыты?
        # эвристика: сначала слова, покрывающие самую дефицитную букву ответа
        for w in candidates_covering_rarest(remaining_need, pool):
            if shares_letter(w, answer):         # инвариант «≥1 буква» (предчек)
                solution.append(w)
                if bt(remaining_need - contribution(w), words_left - 1):
                    return True
                solution.pop()
        return False
    ...
```

**Фаза B — назначить кружки.** Как только K слов выбраны — распределить, какие именно позиции
обводить, — это **уже решённая задача**: `redistribute_circles.py` (backtracking по мультисету
букв ответа с уклоном в наименее загруженное слово → каждое слово задействовано, распределение
1–3/слово). Переиспользуем как есть.

### Почему решений много — proof-of-concept с реальными числами

`poc_word_search.py` строит banded-индекс и рандомизированной выборкой (20 000 сэмплов на кейс)
оценивает **число различных K-словных покрытий**, удовлетворяющих every-word-инварианту.
Прогон на ноутбуке:

```
answer       K len  band   pool  hits/20k     est.coverings    samples/s
--------------------------------------------------------------------------
NAP          1   4   top    259       149                 2        505,289
MEET         2   4   mid    615       211             1,992        411,557
HEIRFORCE    2   5   low   2406         1               145        378,544
SIRPRIZE     4   5   low   2406        37     2,576,673,699        277,381
PANERELIEF   5   6  rare   5256      11    1,331,890,031,018,604,215   229,236
```

Читается так:
- **Пространство огромно.** Даже самый злой уровень `PANE RELIEF` (5 слов, редкий банд) имеет
  порядка **10¹⁸ валидных покрытий** — генератор не голодает по контенту. Простые уровни ограничены
  сверху не пространством, а короткостью ответа (`NAP` — всего 2 варианта, и это нормально: для L1
  разнообразие не нужно).
- **Скорость.** Рандом-поиск идёт **230k–500k сэмплов/сек**. Настоящий решатель — не рандом,
  а backtracking с эвристикой «дефицитная буква вперёд», он на порядки эффективнее рандома
  и находит валидный набор за миллисекунды.
- **Честная тонкость.** `HEIRFORCE` дал лишь 1 попадание на 20k рандом-сэмплов — узкий кейс
  (9-буквенный ответ на 2 слова по 5 → буквы впритык). **Ровно поэтому production-решатель —
  backtracking, а не рандом:** рандом плохо ловит тесные покрытия, backtracking с эвристикой
  находит их гарантированно, если решение существует. PoC-рандом здесь — консервативная нижняя
  оценка; реальная плотность выше.

**Выход шага:** для уровня — `K` слов + распределение кружков (мультисет кружков == буквы ответа,
каждое слово ≥1). Пропускная способность: **тысячи уровней/час** на одном ядре.

## 4. Генерация скрэмблов под целевой скрэмбл-скор P4

Вход: слово `answer` + целевой P4 (0–1) из строки таблицы. Формула P4 — из `levels-design §3`
(та же, что в `validate_levels.py`): доля разрушенных биграмм минус штрафы за первую/последнюю
букву на месте.

Алгоритм — **rejection sampling с направленным мутатором** (P4 монотонен по «разрушенности»,
поэтому подобрать перестановку под цель легко):

```
make_scramble(answer, target_p4, tol=0.07):
    for _ in range(200):
        s = shuffle(answer)                      # случайная перестановка
        # для лёгких уровней: сохранить первую букву / биграмму (тянет P4 вниз)
        # для сложных: максимизировать разрушение биграмм (тянет P4 вверх)
        if scramble_score(answer, s) within tol of target_p4
           and s != answer
           and zipf(s) < 3.0                      # гейт (b): не настоящее слово
           and s.lower() not in PROFANITY:        # гейт (f) на самом скрэмбле
            return s
    # fallback: взять лучший из просмотренных по |p4 - target|
```

Тонкости из `levels-design §3`, которые алгоритм соблюдает:
- **Развороты слова не годятся** для сложных уровней: `WEARY→YRAEW` формула честно даёт P4≈0
  (обратное соседство = «сохранённая» биграмма), поэтому на поздних уровнях они не пройдут по цели.
- **Псевдослова поощряются косвенно:** чтобы разрушить биграммы, скрэмбл получает произносимый,
  но не-словарный вид (`ESRPRSUE`, `MEHBUL`) — и это же требует гейт (b).

Дёшево: сотни перестановок/слово × миллисекунды. **Тысячи слов/сек.**

**Выход:** по скрэмблу на каждое слово, P4 в пределах допуска от цели.

## 5. Автовалидация — `validate_levels.py` как гейт (включая неочевидные кейсы)

Сгенерированный уровень **не публикуется**, пока не пройдёт все проверки. Гейт уже написан
(`validate_levels.py`), генератор его переиспользует пофайлово:

| Гейт | Что ловит | Неочевидность |
|---|---|---|
| (a) | скрэмбл — перестановка ответа | — |
| (b) | скрэмбл ≠ ответ И не словарное слово (zipf<3) | — |
| (c) | мультисет кружков == буквы ответа | — |
| (d) | ответ — реальное слово (zipf>2.5) | — |
| (e) | difficulty монотонен (с дипами на id 5/8) | глобальная проверка по кривой |
| (f) | **профанити в скрэмбле** | **случайная перестановка может сложиться в слэр — проверяем и её, не только ответ** |
| (g) | каждое слово отдаёт ≥1 кружок | authentic-Jumble инвариант |

Два кейса, которые бриф просил не забыть:

1. **Вторая анаграмма / единственность.** Для скрэмбла надо проверять, что нет *другого*
   правдоподобного словарного решения того же банда (иначе игрок введёт валидное слово и не
   зачтётся). Расширение к (b): для мультисета букв скрэмбла собрать все анаграммы из словаря
   с zipf ≥ порога банда; если их >1 — либо перегенерить скрэмбл, либо **осознанно оставить
   как ловушку** (P5=1) и пометить `trap: true`. Ловушка — фича поздних уровней, но она должна
   быть *решением*, а не случайностью, поэтому единственность проверяется всегда.
2. **Профанити в СЛУЧАЙНОМ скрэмбле** — гейт (f) уже на месте: `shuffle("ARSENAL")` может выдать
   что-то нехорошее. Проверяется и на скрэмблах, и на любых сгенерированных строках, а не только
   на кураторском ответе.

Плюс **дедуп уровней**: нормализованный ключ (отсортированные ответы + surprise) против уже
выпущенного каталога.

Валидация детерминированная и быстрая: **тысячи уровней/сек** (единственная цена — вызовы
`zipf_frequency`, они кэшируются).

**Выход:** уровень со статусом PASS/FAIL + причины. FAIL → авто-регенерация затронутого шага
(обычно скрэмбла или замена слова), не всего уровня.

## 6. Картинки-карикатуры — вторичный боттлнек (деньги + ревью)

Вход: `answer` + `setup_caption` + `pun_word` + P8 (прозрачность из таблицы).

```
answer+caption+P8 ──LLM──▶ image-промпт ──image-API батч──▶ картинка ──vision-check──▶ годная/брак
```

1. **Промпт-шаблон** (стайл-гайд из art-брифа) заполняется LLM: сцена, персонажи, реплики,
   **уровень прозрачности по P8** (P8=5 → сцена почти показывает ответ; P8=1 → лишь косвенный намёк).
2. **Батч в image-API.** Стоимость **~$0.005–0.02/картинка** (§9).
3. **Автопроверка vision-моделью:** нет мусор-текста в картинке (генераторы любят рисовать
   искажённые буквы), сцена соответствует промпту, семейно-безопасно. Брак → регенерация.

Это боттлнек №2 — деньги на API и время на ревью брака — но он **параллелится** (батчи) и
дешевле человеко-модерации каламбуров. Часть уровней (ранние, туториальные) может переиспользовать
шаблонные сцены, снижая нагрузку.

**Выход:** валидная карикатура на уровень, привязанная к ответу и P8.

## 7. Скоринг difficulty_score

Каждый собранный уровень имеет фактические P1–P8 (число слов, длины, Zipf-банды, скрэмбл-скоры,
наличие ловушки, длина/словность ответа, тип каламбура, прозрачность картинки). Скор — взвешенная
сумма (формула-каркас из `levels-design §5`, **точные веса калибрует balance-economist в фазе 5,
`balance-notes`**):

```
score = 0.9·P1 + 0.25·P2 + 0.35·P3 + 1.2·P4 + 0.6·P5 + 0.12·P6
        + w(P7) + 0.30·(6−P8)
где w(direct)=0, w(idiom)=0.3, w(homophone)=0.6, w(double-layer)=1.0
```

**Выход:** `difficulty.score` уровня — число, по которому уровень кладётся в кривую (§8-кривая).

## 8. Раскладка по слотам кривой + ручная QA 10%

**Раскладка (связка с балансом — сердце документа).** Генератор работает **не «сгенерь случайное
и посмотри куда попало»**, а **обратно**: берёт из балансовой таблицы фазы 5 (100 строк, каждая —
целевой `difficulty.score` и целевые P1–P8) конкретный слот и **генерит уровень ПОД эти параметры**.

```
для каждого слота v в таблице 100 уровней:
    target = balance_table[v]        # {score, P1..P8}  ← кривая-пила фазы 5
    answer = pick_from_pun_bank(pun_type=target.P7, len≈target.P6)
    words  = select_words(answer, K=target.P1, len=target.P2, band=target.P3)   # §3 CSP
    scrambles = [make_scramble(w, target.P4) for w in words]                    # §4
    circles = redistribute_circles(words, answer)                              # §3B
    image  = gen_cartoon(answer, caption, P8=target.P8)                        # §6
    level  = assemble(...)
    assert validate(level) == PASS                                             # §5
    assert abs(level.score - target.score) < 0.3   # попали в слот кривой?
```

Так пила/передышки из `levels-design §4` (спайки на 4/7/10, дипы на 5/8, а в полной кривой —
их 100-уровневый аналог) **воспроизводятся по построению**: параметры берутся из таблицы, а не
получаются случайно. Уровень, чей фактический score отклонился от целевого >0.3, отбраковывается
и регенерится — так каждый слот кривой гарантированно заполнен уровнем нужной сложности.
Инвариант якорных точек (`levels-design §5`: v1→1.0, v34 спайк, v45/v78 дипы, v100→6.8) —
жёсткое ограничение, которое раскладка обязана соблюсти.

**Ручная QA 10%.** Из каждой партии человек проверяет случайные 10% по критериям: решаемо ли
за целевое время? смешно ли? картинка намекает, но не спойлерит? Провал сэмпла → правило фильтра
дорабатывается, партия перегенерится. 10% — компромисс между уверенностью и стоимостью
(остальные 90% защищены детерминированным гейтом §5).

**Выход:** каталог уровней, разложенный по кривой, 100% прошли автогейт, 10% прошли человека.

## 9. Числа: пропускная способность и стоимость

### Пропускная способность по шагам (одно ядро ноутбука)

| Шаг | Темп | Боттлнек? |
|---|---|---|
| Индекс корпусов | 0.29 с (один раз) | нет |
| Подбор слов (CSP §3) | тысячи уровней/час | нет |
| Скрэмблы (§4) | тысячи слов/сек | нет |
| Валидация (§5) | тысячи уровней/сек | нет |
| **Каламбуры (§2, модерация)** | **~40 одобренных/час/человек** | **ДА №1** |
| **Картинки (§6)** | батч, ~секунды/шт + ревью брака | боттлнек №2 (деньги+ревью) |

Автоматическая часть (слова+скрэмблы+валидация) — **не ограничение**: она выпустила бы десятки
тысяч скелетов уровней в час. Реальный потолок задают два человеко/деньго-зависимых шага.

### Итог: сколько уровней/день силами 1 человека + автоматика

Расчёт при 6 продуктивных часах модерации в день:

- **Банк каламбуров:** 40 одобренных/час × 6 ч = **240 новых каламбуров/день**.
  Но каламбур — переиспользуемый актив, и один каламбур = один уровень (ответ уникален).
- **QA-сэмпл 10%:** если выпускаем N уровней/день, человек смотрит 0.1·N. При N=100 это 10 уровней
  ≈ 0.5 ч — влезает в тот же день.
- **Картинки:** генерация в облаке параллельна, человеку остаётся ревью брака vision-модели
  (~5–10% партии).

**Связывающее звено — модерация.** Если каждый уровень требует свежего одобренного каламбура,
потолок = 240/день теоретически, но с учётом QA-сэмпла и ревью картинок реалистичный устойчивый
темп —

> **≈ 100 уровней/день силами 1 дизайнера-модератора + автоматика.**

Расчёт 100/день: ~2.5 ч на модерацию каламбуров (100 шт при 40/ч), ~0.5 ч QA-сэмпла (10 шт),
~1 ч ревью картинок-брака, ~2 ч буфер (правки промптов, отбраковка, оркестрация). ≈ 6 ч/день.
Автоматика (слова/скрэмблы/валидация/раскладка) в этот бюджет не входит — она фоновая и мгновенная.

**Полная игра (100 уровней) → ~1 неделя работы 1 человека** (с запасом на итерации и перегенерацию).
**Второй модератор ≈ удваивает темп** до ~200/день — узкое место масштабируется линейно по людям.

### Стоимость на 1000 уровней

| Статья | Расчёт | Стоимость |
|---|---|---|
| **LLM: генерация каламбуров** | ~1000 ответов, батчами по 40; ~25 вызовов × ~2k токенов вх/вых. Даже по премиум-тарифу ~$3–15/1M токенов | **~$1–5** |
| **LLM: генерация image-промптов** | 1000 промптов × ~500 токенов | **~$1–3** |
| **LLM: авто-фильтры/vision-проверки** | 1000 vision-вызовов (мусор-текст) | **~$5–15** |
| **Image API** | 1000 картинок × $0.005–0.02, +~10% на брак/регенерацию | **~$5.50–22** |
| **Модерация каламбуров** | 1000 / 40 в час = 25 ч × ставка дизайнера (напр. $30/ч) | **~$750** |
| **QA-сэмпл 10%** | 100 уровней × ~3 мин = 5 ч × $30 | **~$150** |
| **Ревью брака картинок** | ~2–3 ч × $30 | **~$60–90** |
| **ИТОГО на 1000 уровней** | | **≈ $970–1035, из них ~$960 — человеко-часы, <$50 — вся автоматика/API** |

**Главный вывод по деньгам:** compute/API-часть **пренебрежимо мала (<$50/1000)**. ~95% стоимости —
человеко-время модерации и QA. Значит, **инвестиция в качество автофильтров** (чтобы человек
меньше отсматривал мусора) — прямой рычаг стоимости: улучшил фильтр с 30% до 50% выхода годного —
срезал модерацию почти вдвое.

## 10. Как генератор попадает в целевую кривую (сводка связки с балансом)

1. Балансовая таблица фазы 5 (`balance-notes`) — **источник истины**: 100 строк, каждая задаёт
   целевой `difficulty.score` и целевые P1–P8 (пила со спайками/дипами, якоря v1/v34/v45/v78/v100).
2. Генератор идёт **по слотам**, а не наугад: для слота берёт `target` и генерит уровень под него
   (тип каламбура ← P7, число/длина/банд слов ← P1/P2/P3, скрэмбл-скор ← P4, прозрачность ← P8).
3. Фактический score уровня считается той же формулой (§7). Отклонение >0.3 от целевого → отбраковка
   и регенерация. Так кривая-пила воспроизводится **по построению**, а не проверяется постфактум.
4. Веса формулы score — единый контракт `levels-design`↔`balance-notes`; калибрует их
   balance-economist в фазе 5. Генератор их только применяет.
