# Бриф: гибридная монетизация Scroll Puzzle

Исполнитель: monetization-producer.
Deliverable: `deliverables/monetization/scroll-puzzle-hybrid.md` (+ `hybrid-model.xlsx` для расчёта).
База фактов: `01-research/scroll-puzzle-research.md` — НЕ выдумывать факты об игре, чего нет — помечать допущением.

Структура ответа = структура вопросов задания (a→d), ревьюеру должно быть легко сверять.

## a) Гипотезы перевода на гибрид + внедрение + последовательность апдейтов

Опорные факты research: interstitial с ~14 уровня и посреди уровня — главная жалоба игроков;
мета — только коллекции; coins существуют, но тратить некуда; у издателя есть успешный кейс
гибрида 50/50 (Alien Invasion, rewarded-only). Кандидаты в гипотезы (выбрать и оформить 2–4,
каждая: формулировка «если…, то…, потому что…»):

- **H1 «Экономика бустеров»**: ввести трату coins — бустеры (подсказка «где пара», авто-сбор
  фрагмента, undo) → появляется спрос на coins → IAP-паки монет + rewarded «монеты×2 после уровня»
  и «бустер за просмотр». Ядро гибрида: реклама и IAP кормят одну экономику.
- **H2 «Remove Ads + смягчение interstitial»**: SKU remove ads ($4.99, сохраняет rewarded) +
  убрать interstitial из середины уровня (главная жалоба) и капнуть частоту после уровней.
  Гипотеза: retention вырастет настолько, что LTV не упадёт, а конверсия в первый платёж вырастет.
- **H3 «Оффер-слой»**: starter pack (однораз., дешёвый, монеты+бустеры+N дней без interstitial),
  piggy bank (копилка монет за прогресс, выкуп за $), later — цепочка офферов по сегментам.
- **H4 «Платная мета коллекций»**: премиум-коллекции/темы, «сундук коллекционера» —
  монетизация единственной существующей меты. (Осторожно: дальше от ядра, дороже в разработке.)

Последовательность апдейтов — обосновать зависимостями: экономика (H1) — фундамент, без неё
IAP нечего продавать; H2 можно параллельно (дёшево); H3 после H1 (офферам нужен контент экономики);
H4 — по результатам. Для каждого апдейта: состав, длительность, способ теста (A/B на % аудитории,
контрольная группа, критерии успеха/отката, длительность теста ≥2 недели на когорту).

## b) Постановка задачи команде

Формат «одна страница на апдейт», не ТЗ. Скелет: **Зачем** (проблема цифрами) → **Что делаем**
(3–7 буллетов скоупа, что НЕ делаем) → **Как поймём, что получилось** (2–4 метрики с целевыми
значениями и сроком) → **Кому что** (GD — конфиг цен, dev — фичи+ивенты, UI — экраны, QA — кейсы,
аналитик — дашборд A/B). Написать реальный текст такой постановки для первого апдейта (не мета-описание).

## c) Метрики и ивенты

Два уровня:
- **Ивенты** (что логируем): ad_impression {placement, type}, ad_offer_shown/accepted (opt-in rate
  rewarded), iap_shown/initiated/completed {sku, price}, booster_used {type, level}, coins_earned/spent
  {source/sink}, level_start/complete/fail {duration, boosters}, session_start {number, length},
  piggy_full/piggy_purchase, churn-маркеры.
- **Метрики** (что смотрим): ARPDAU total = ad + IAP (следить за каннибализацией: ad-ARPDAU
  просядет — общий должен вырасти), conversion to payer %, repeat purchase %, rewarded opt-in %,
  interstitial per DAU, retention D1/D7/D30, session length/count, уровни/DAU, sink/source ratio
  экономики, LTV по когортам.
- Для каждой гипотезы: ожидаемое направление и величина эффекта + guardrail-метрики
  (retention не ниже −1 п.п., crash-free, скорость прохождения).

## d) Расчёт на выдуманных данных

Модель «до/после» в xlsx + выжимка в markdown. Входные допущения (выдуманные, но правдоподобные,
масштаб взять крупнее текущего софт-ланча — например DAU 50–100K после разгона UA):
DAU, retention, interstitial/DAU × eCPM ($8–12 Tier-1/бленд), rewarded/DAU × eCPM ($15–25),
конверсия в платящих 1.5–2.5%, средний чек, доли SKU. Посчитать: ARPDAU до (ads-only) →
после каждого апдейта (штук 3 сценария: песс./база/опт.), помесячный эффект на revenue,
чувствительность к каннибализации interstitial. Арифметика обязана сходиться и быть воспроизводимой
(формулы в xlsx, а не числа из головы).
